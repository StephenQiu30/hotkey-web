# 内容归档阅读与工作台可视化计划

## 1. 目标与开工门禁

本计划只修改 `hotkey-web`。完成后，用户可从事件证据和采集内容进入系统内详情，读取服务端授权归档的 Markdown，并通过浏览器“打印 / 保存 PDF”；管理员可在创建来源时显式开启 `allow_body_storage`；工作台不再让证据时间线、窄图表和 460px 侧栏形成三列竞争。

开工前必须同时满足：

1. Design-001 与 PRD-002 已审核，本文档 `status: approved`、`review_status: approved`、`execution_status: ready`。
2. `hotkey-server` 的 PLAN-019 已完成，运行中的 `/openapi.json` 包含 `GET /api/v1/contents/{id}/document`。
3. `npm run openapi:generate` 可从本地服务生成客户端；不得手写 `HotKeyAPI` 类型或请求函数。
4. 所有新增测试位于仓库根目录 `test/`，测试先于生产实现提交。

## 3. Task 1：冻结并生成 OpenAPI 客户端

**测试文件：** 修改 `test/unit/lib/openapi-generation.test.ts`。

先增加断言：生成目录必须存在 `getContentsIdDocument`，其返回类型引用生成的内容文档 DTO；`src/` 中不得出现手写 `/api/v1/contents/.../document` 请求、重复 `ContentDocument` interface 或其他服务目录。

RED：

```bash
npm run test:unit -- test/unit/lib/openapi-generation.test.ts
```

预期失败信号为 `contents.ts` 不包含 `getContentsIdDocument`，或生成类型缺少 `ready | not_captured` 契约；不能用放宽断言消除失败。

最小实现：启动已包含 PLAN-019 的后端，运行 `npm run openapi:generate`。仅提交生成器实际改动的 `contents.ts`、`typings.d.ts` 和可选 `index.ts`，保持 `openapi2ts.config.ts` 使用官方 `@umijs/openapi` 配置和共享 Axios adapter。

GREEN：

```bash
npm run openapi:generate
npm run test:unit -- test/unit/lib/openapi-generation.test.ts
npm run typecheck
```

通过标准：客户端能从生成类型区分 `ready`、`not_captured`；503 仍走共享请求错误，不在页面伪造第三种成功状态。

## 4. Task 2：内容详情、安全 Markdown 与打印

**测试文件：** 创建 `test/unit/components/dashboard/ContentDocumentViewer.test.tsx`、`test/unit/app/dashboard-content-detail.test.tsx`。

RED 场景：

1. `ready` 展示标题、来源、时间、内容范围提示、Markdown、系统内原站入口和“打印 / 保存 PDF”。
2. GFM 表格、列表、代码块可读；HTML 字符串不作为 DOM 执行；外链带 `target="_blank"` 和 `rel="noreferrer"`。
3. 点击打印调用一次 `window.print()`；打印按钮在 `not_captured` 时不可用。
4. `not_captured` 明确说明“本条未归档正文/摘要”，不把标题或 canonical URL 伪造成正文；仍可访问原站。
5. 404 显示内容不存在；403 显示无权访问并提供安全返回入口，且不保留或泄露此前正文；503 显示归档暂不可用并提供重试。任何错误正文都不直接注入页面。
6. 动态路由只调用生成的 `getContentsIdDocument`，非法 id 不发请求。

RED：

```bash
npm run test:unit -- \
  test/unit/components/dashboard/ContentDocumentViewer.test.tsx \
  test/unit/app/dashboard-content-detail.test.tsx
```

预期失败信号为组件或动态路由不存在、打印回调未调用、原始 HTML 被渲染或状态空态缺失。

最小实现：

- `ContentDocumentViewer` 复用 `react-markdown` 与 `remark-gfm`，不添加 `rehype-raw`、HTML parser 或第二套 Markdown 库。
- 动态页通过生成服务读取 document；使用现有 `.app-page`、`.panel`、`Button`、`Badge` 和 lucide 图标。
- 阅读正文宽度约 `72ch`；表格容器可水平滚动，代码和长链接不得撑破 viewport。
- `src/app/globals.css` 增加 `@media print`：隐藏顶部导航、toast、操作条和装饰边框，只保留标题、来源、canonical URL、时间、范围声明和正文。

GREEN：

```bash
npm run test:unit -- \
  test/unit/components/dashboard/ContentDocumentViewer.test.tsx \
  test/unit/app/dashboard-content-detail.test.tsx
npm run typecheck
```

## 5. Task 3：管理员来源正文保存开关

**测试文件：** 创建 `test/unit/app/dashboard-sources-page.test.tsx`。

RED 场景：

1. admin 打开“新增来源”时看见“保存来源正文/摘要用于归档预览”，初始未勾选。
2. 默认创建请求提交 `config.allow_body_storage=false`；只有显式勾选才提交 `true`。
3. 成功创建、关闭弹窗或切换来源类型后，开关重置为 `false`，不能把上一次授权带到下一次创建。
4. viewer/editor 看不到新增按钮和开关，不能从 UI 触发创建。
5. 说明文案明确“只保存来源 Feed 实际提供的正文/摘要，不抓取原网页；启用前确认来源条款”。

RED：

```bash
npm run test:unit -- test/unit/app/dashboard-sources-page.test.tsx
```

预期失败信号为开关不存在、默认请求没有明确 `false`、非 admin 可见或状态未重置。

最小实现：修改 `src/app/dashboard/sources/page.tsx`，复用现有 `Checkbox`，把 `allow_body_storage` 纳入本地 form 并映射到生成的 `CreateSourceRequest.config`。不创建前端临时类型，不允许修改已发布来源的归档许可。

GREEN：

```bash
npm run test:unit -- test/unit/app/dashboard-sources-page.test.tsx
npm run typecheck
```

## 6. Task 4：工作台证据与热度降密度

**测试文件：** 创建 `test/unit/components/dashboard/EventEvidenceTimeline.test.tsx`、`test/unit/components/dashboard/EventHeatComparison.test.tsx`；修改 `test/unit/app/dashboard-page.test.tsx`。

RED 场景：

1. “证据” Tab 只包含全宽证据时间线，不再包含 240px 图表；每条证据都有系统内“阅读归档”和独立“访问原站”入口。
2. 页面展示“当前展示 n / 共 m 条”。`n` 来自实际成功加载的 Content，`m` 来自事件成员响应，不用 `heat.content_count` 或假数字代替失败的详情请求。
3. 详情加载保留失败项的计数事实，并对“部分证据暂不可读”给出说明；不把 `Promise.allSettled` 失败静默成全部成功。
4. “事件热度对比”只在“信号” Tab 的全宽区出现；全部热度为 0 或缺失时展示“热度尚未计算”，不挂载残缺 Recharts 坐标。
5. 桌面主区/情报区为 `minmax(0,1fr) 360px`；窄于大屏断点后单列，证据、按钮和标题不横向溢出。
6. “关联报告”在没有 event-report 关联字段时改为“最近报告”，避免把任意首条报告伪装为事件关联。

RED：

```bash
npm run test:unit -- \
  test/unit/components/dashboard/EventEvidenceTimeline.test.tsx \
  test/unit/components/dashboard/EventHeatComparison.test.tsx \
  test/unit/app/dashboard-page.test.tsx
```

预期失败信号为新组件不存在、图表仍在证据 Tab、证据仍静默截为 8 条且无 n/m、零值仍渲染坐标或旧“关联报告”文案仍存在。

最小实现：从 `src/app/dashboard/page.tsx` 提取两个职责单一组件；保留后端当前一次返回的成员总数，并只对有 `content_id` 的成员读取详情。若仍以 8 条限制保护并发，必须把限制作为 UI 事实展示，不能写成全部证据。主区/侧栏断点和宽度只在现有 Tailwind 类中调整，不引入新布局库。

GREEN：

```bash
npm run test:unit -- \
  test/unit/components/dashboard/EventEvidenceTimeline.test.tsx \
  test/unit/components/dashboard/EventHeatComparison.test.tsx \
  test/unit/app/dashboard-page.test.tsx
npm run typecheck
```

## 9. 提交边界与回滚

按 `AGENTS.md` 使用中文提交并保持测试先行：

1. `test:` 提交各 Task 的 RED 测试。
2. `impl:` 提交最小页面、组件、表单和样式实现。
3. 如有纯行为不变拆分，再提交 `refactor:`。
4. `chore:` 最后提交 `@umijs/openapi` 生成文件和文档状态；生成文件禁止手改。

在最终组合提交前，生成文件可留在工作树供 TypeScript 编译，但不得宣称中间提交可单独发布。回滚只回退 Web 页面和生成客户端，不修改后端 Schema、MinIO 证据或来源授权事实。若后端 document 契约尚未 accepted，整个 Plan 保持 backlog，禁止用手写 mock API 提前上线。

## 10. 残余风险

- 浏览器打印结果受浏览器、字体和用户打印设置影响，自动测试只能验证 print CSS 与打印调用，不能替用户确认系统保存对话框。
- `not_captured` 可能代表来源未提供 body、授权关闭或历史记录无 asset；Web 只能展示服务端事实，不推断具体原因。
- 当前事件成员接口不分页；若后续增加 cursor/total，n/m 数据来源必须随 OpenAPI 重新设计，不能在前端估算。
