## 4. RED 证据要求

实施前必须至少记录以下真实失败，不接受删除断言或把接口写成手工类型：

```bash
npm run test:unit -- test/unit/lib/openapi-generation.test.ts
npm run test:unit -- \
  test/unit/components/dashboard/ContentDocumentViewer.test.tsx \
  test/unit/app/dashboard-content-detail.test.tsx
npm run test:unit -- test/unit/app/dashboard-sources-page.test.tsx
npm run test:unit -- \
  test/unit/components/dashboard/EventEvidenceTimeline.test.tsx \
  test/unit/components/dashboard/EventHeatComparison.test.tsx \
  test/unit/app/dashboard-page.test.tsx
npm run test:unit -- test/unit/components/dashboard/CollectionWorkspace.test.tsx
```

预期 RED 信号分别为：document 生成函数缺失；详情/预览组件缺失；`allow_body_storage` 开关或请求映射缺失；图表仍挤在证据 Tab、n/m 与零值空态缺失；采集内容缺少内部详情链接或分栏。

实际失败摘要：OpenAPI RED 缺少 `getContentsIdDocument`；详情、证据和热度组件 RED 为模块不存在；来源 RED 缺少授权 checkbox；采集分栏 RED 缺少 `collection-pipeline` 与内部阅读入口。随后仅通过服务端生成器和对应最小实现转绿，未手写 document 请求或 DTO。

## 6. 浏览器验收步骤

1. 使用 admin 登录并打开来源管理，确认开关默认关闭；分别创建关闭和开启归档的可丢弃 RSS 来源，Network body 中 `config.allow_body_storage` 分别为 `false`/`true`。
2. 以 viewer 打开 `/dashboard/contents`，从已归档 Content 的“阅读归档”进入 `/dashboard/contents/{id}`；确认 Markdown、来源、范围声明、原站链接和“打印 / 保存 PDF”可见。
3. 打开未捕获 Content，确认没有正文和可用打印按钮；页面不得把标题、摘要猜测或 canonical URL 当作 Markdown。
4. 使用受控权限 fixture 验证 403 显示“无权访问”和安全返回入口，且页面不保留此前正文；再临时停止测试 MinIO 或使用受控故障 fixture，确认 503 显示“归档暂不可用”和重试，恢复后同一路由可重新读取。
5. 进入打印预览，确认导航、toast、按钮和装饰隐藏，标题、来源、canonical URL、发布时间、内容范围声明与正文保留；由人工在系统对话框确认可保存 PDF。
6. 在 1440×900、1024×768、390×844 验证 `/dashboard` 与 `/dashboard/contents`：无横向滚动，侧栏在窄屏下顺序堆叠，证据不再与 240px 图表并列，零热度为空态，n/m 与 API 一致。
7. 清理可丢弃来源、测试账号和浏览器会话；不删除合法历史 Content 或真实来源。

## 7. 安全与事实核对

- [x] 页面不启用 `rehype-raw`，单元测试确认原始 HTML 不会作为 DOM 执行。
- [x] 外链使用新窗口与 `noreferrer`，内部链接不伪装成原站。
- [x] Web 不持久化正文、object key、MinIO endpoint 或认证信息。
- [x] `not_captured` 不推断是授权关闭、Feed 缺失还是历史数据。
- [x] “打印 / 保存 PDF”只描述浏览器能力，不宣称服务端已生成 PDF。
- [x] “最近报告”不再被标记为与当前事件已关联。

## 8. 未执行项与残余风险

真实 admin、ready/not_captured、404、受控 MinIO 503 与恢复、工作台/采集内容双入口均已执行。未单独保存 1440×900 和 390×844 真实截图，响应式风险由 `.app-page`、`xl` sticky/双列、小屏 static/单列的精确测试兜底；系统保存 PDF 对话框受操作系统控制，最终文件保存仍需用户确认，自动化验证 `window.print()` 调用、print CSS 与可打印内容结构。因此结论为 `accepted_with_risk`，不伪记为无风险 accepted。
